Create `ItineraryItemService.java` with:

**Dependencies (constructor injection):**
- TripRepository tripRepo
- ItineraryDayRepository dayRepo
- ItineraryItemRepository itemRepo

**Methods:**

1. `addItem(UUID tripId, UUID dayId, String userId, String destinationRef, LocalTime timeSlot, String note, String photoUrl) → ItineraryItem`
   - Verify trip ownership: `tripRepo.findByIdAndUserId(tripId, UUID.fromString(userId))` → TripNotFoundException
   - Verify day belongs to trip: day from `dayRepo.findById(dayId)` must have `tripId` matching → DayNotInTripException
   - Acquire lock: `itemRepo.findByDayIdForUpdate(dayId)` (per D-03, serializes writes)
   - Compute position: `itemRepo.findMaxPositionByDayId(dayId).orElse(0) + 100`
   - Sanitize note: `note != null ? Jsoup.clean(note, Safelist.none()) : null` (per D-04/D-05)
   - Create entity: `new ItineraryItem(UUID.randomUUID(), dayId, destinationRef, position, timeSlot, sanitizedNote, photoUrl)`
   - Save and return
   - Annotate: `@Transactional(isolation = Isolation.REPEATABLE_READ)`

2. `updateItem(UUID tripId, UUID itemId, String userId, Integer newPosition, UUID newDayId, LocalTime timeSlot, String note, String photoUrl) → ItineraryItem`
   - Verify trip ownership (same as above)
   - Load all days for this trip: `dayRepo.findByTripIdOrderByDayIndex(tripId)`
   - Find current item by id (from any day of this trip) — if not found → ItemNotFoundException
   - Verify item belongs to a day in this trip → ItemNotFoundException
   - If newDayId != null and differs from current: validate newDayId is in trip's days → DayNotInTripException; update item.itineraryDayId
   - Acquire lock on target day: `itemRepo.findByDayIdForUpdate(effectiveDayId)`
   - If newPosition != null: check gap. If gap < 2 between neighbors at target position → call `reindexDay(effectiveDayId)` first, then set position. Otherwise just set position.
   - If note != null: sanitize with Jsoup.clean(note, Safelist.none()), set on item
   - If timeSlot provided (use a sentinel or wrapper to distinguish null="clear" from absent="no change"): update timeSlot. For simplicity, if UpdateItemRequest has a field present (non-null wrapper) → update. Use the record's field directly — null means "no change" per PATCH semantics. To explicitly clear timeSlot, frontend sends `"timeSlot": null` — but since Java record field is already null... Use a `boolean timeSlotPresent` flag or just accept that null = no change and require a special value to clear. **Simplification:** treat `timeSlot` field in UpdateItemRequest as nullable — null means no change. To clear, accept empty string "" and map to null in service. Same pattern for note.
   - If photoUrl != null: set on item
   - Set updatedAt = Instant.now()
   - Save and return
   - Annotate: `@Transactional(isolation = Isolation.REPEATABLE_READ)`

3. `deleteItem(UUID tripId, UUID itemId, String userId) → void`
   - Verify trip ownership
   - Find item, verify it belongs to a day of this trip → ItemNotFoundException
   - `itemRepo.delete(item)`
   - Annotate: `@Transactional`

4. `private void reindexDay(UUID dayId)`
   - Load all items for day ordered by position: `itemRepo.findByItineraryDayIdOrderByPositionAsc(dayId)`
   - Renumber: position = (index + 1) * 100
   - For each: `itemRepo.updatePosition(item.getId(), newPosition)`
   - Per D-02: single SQL window function equivalent — batch update via individual updatePosition calls (JPA constraint). Alternative: use a native @Query with window function `ROW_NUMBER() OVER (ORDER BY position) * 100`. **Use the native query approach:**
     ```java
     @Modifying @Query(value = "UPDATE trip.itinerary_items SET position = sub.new_pos, updated_at = NOW() FROM (SELECT id, (ROW_NUMBER() OVER (ORDER BY position)) * 100 AS new_pos FROM trip.itinerary_items WHERE itinerary_day_id = :dayId) sub WHERE trip.itinerary_items.id = sub.id", nativeQuery = true)
     void reindexDay(@Param("dayId") UUID dayId);
     ```
     Add this to ItineraryItemRepository (include in this task since it's tightly coupled to the service). The executor should add `reindexDay` to the repository AND use it in the service.

**Import:** `org.jsoup.Jsoup` and `org.jsoup.safety.Safelist`

**Key pattern:** Every mutating method first verifies ownership (tripRepo.findByIdAndUserId), then verifies day/item belongs to trip, then acquires lock, then operates.
