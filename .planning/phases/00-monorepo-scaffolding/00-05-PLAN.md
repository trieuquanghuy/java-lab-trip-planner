<objective>
Build the eureka-server skeleton — the service registry every other backend service registers with at startup. Eureka is the simplest of the 5 services: no DB, no real observability lib (registry traffic not in the trace path), no JWT, no domain. It exists to host the dashboard at `:8761` (per ROADMAP SC #2) and to give downstream services somewhere to publish their lease.

Purpose: Without eureka-server running, the gateway and 3 DB-backed services in Wave 3 will fail their `EurekaClient` startup and stay un-registered. SC #2 (4 services in Eureka) cannot be met without this plan completing first.

Output: A buildable Spring Boot 3.5.x service that starts on port 8761, displays the Eureka dashboard, and accepts lease registrations from the other 4 services that will come online in subsequent waves.
</objective>

    2. **`logback-spring.xml`** verbatim from PATTERNS.md Bucket C (line 415-423):
       ```xml

       <configuration>
           <include resource="logback-spring-base.xml" />
       </configuration>
       ```
       Even though eureka-server does NOT depend on `libs/observability` (per Task 5.1), the `logback-spring-base.xml` is on the runtime classpath only when the lib is a dep. **Therefore for eureka-server we have a choice**:
       - **Option A** (chosen): Skip the include and rely on Spring Boot's default Logback config. The dashboard is the primary user surface; structured JSON logs are a Phase 1+ concern for cross-service correlation. Write the file as a pass-through:
         ```xml
         <configuration>
             <!-- eureka-server uses Spring Boot's default Logback config.
                  Other services pull in libs/observability's logback-spring-base.xml include
                  for unified JSON output + MDC (traceId, spanId). Eureka is not in the trace path. -->
         </configuration>
         ```
       - **Option B** (alternative): Add `implementation(project(":libs:observability"))` to Task 5.1 and use the full include. Adds ~5MB to the eureka jar and introduces the autoconfig but produces consistent log shape.

       **Pick Option A** — minimal surface for the registry. Document the rationale in the file's comment. If a future phase needs unified Eureka log format, swap to Option B by adding the lib dep + uncommenting the include.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| network ↔ Eureka dashboard at :8761 | Dashboard exposes registered service names + IPs to anyone who can reach the port |
| client services ↔ Eureka REST API | Lease registration uses no auth in dev (free tier; production-only concern) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-00-16 | Information Disclosure | Eureka dashboard at :8761 leaks internal service hostnames/IPs to anyone on the LAN | LOW | mitigate | D-22 binds eureka to `127.0.0.1:8761` in compose — only the developer's host can reach it. Production deployment is out of scope (v1 = local only). |
| T-00-17 | Spoofing | Malicious "fake auth-service" registers with Eureka and gateway routes traffic to it (Pitfall 1 variant) | MEDIUM | accept | Phase 0 gateway uses **static URIs** (D-02), not `lb://` Eureka-routed URIs — so the registry contents do NOT influence routing in Phase 0. Phase 10 hardening adds mTLS or token-based Eureka client auth before switching to `lb://` URIs. |
| T-00-18 | DoS | Self-preservation disabled means stale leases are evicted aggressively → potential transient 503s | LOW | accept | `enable-self-preservation: false` is a dev-only choice that makes the Eureka behavior more predictable in compose. D-21's 5s lease intervals and 10s expiration are fast enough that a normal restart re-registers within Pitfall 10's <60s budget. |
</threat_model>
