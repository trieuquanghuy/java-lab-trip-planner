## 4. E2E User Flow Verification

### Flow 1: Signup → Verify → Login → Create Trip → Add Item

- Signup: `POST /api/auth/signup` → 201 ✅
- Email: Mailhog receives verification email ✅
- Verify: `GET /api/auth/verify?token=...` → 302 → frontend `/verify?status=success` ✅
- Login: `POST /api/auth/login` → 200 + JWT + cookie ✅
- Create trip: `POST /api/trips` → 201 ✅
- Add item: `POST /api/trips/:id/days/:dayId/items` → 201 ✅

### Flow 2: Search → Browse → Add to Trip

- Search: `GET /api/search?q=london` → city results ✅
- Nearby: `GET /api/destinations?lat=...&lng=...` → attractions ✅
- Detail: `GET /api/destinations/otm:N123` → full detail ✅
- Add: Frontend "Add to Trip" → item endpoint ✅

### Flow 3: Drag-Drop Reorder

- Reorder within day: `PATCH /api/trips/:id/items/:itemId` (position) ✅
- Move between days: `PATCH /api/trips/:id/items/:itemId` (itineraryDayId) ✅
- Optimistic update with rollback on error ✅

### Flow 4: Graceful Degradation

- Circuit breaker on OpenTripMap failure ✅
- Cached results served when provider down ✅
- `providerStatus` communicated to frontend ✅

### Flow 5: Session Persistence

- Login → JWT in memory, refresh in httpOnly cookie ✅
- Page refresh → 401 → auto-refresh via cookie → new access token ✅
- Trips, items, favorites persist across sessions ✅
