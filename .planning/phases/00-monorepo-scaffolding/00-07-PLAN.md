<objective>
Build the three DB-backed Spring Boot service skeletons — `auth-service`, `trip-service`, `destination-service` — using a single shared template instantiated three times via the per-service substitution table from PATTERNS.md Bucket E. Each service has the same shape (~7 files), differing only in name, port, schema, package, env-var prefix, and Flyway history-table name.

Per ROADMAP SC #2 these three services + the gateway register with Eureka. Per ROADMAP SC #5 each service's Flyway V1 baseline runs and produces a `<svc>_flyway_schema_history` table in its own schema. Per ROADMAP SC #3-route, each service hosts `/__health` so the gateway's static-URI routes can forward `/__health/<svc>` calls to it.

Purpose: Lock the per-service-Flyway-history-table convention (Pitfall 3 mitigation in artifact form) and the per-service-DB-user model (D-08) at the service-config level — the actual schemas + users are created by `infra/postgres/init.sql` in Wave 4. By doing the three services in one plan, the executor sees the per-service substitution table inline and the convention drift risk drops to zero.

Output: Three Spring Boot 3.5.x servlet services (NOT reactive — only the gateway is reactive) that each (a) compile, (b) declare `flyway-database-postgresql` runtime dep (Pitfall A), (c) configure their own Flyway history table (Pitfall 3), (d) register with Eureka, (e) host a `/__health` placeholder controller, (f) ship an empty V1 baseline migration.
</objective>
