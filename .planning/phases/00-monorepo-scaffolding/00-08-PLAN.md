       ```sql
       -- Phase 0 scaffold: 3 schemas + 3 service users with schema-scoped grants.
       -- Idempotent: safe to re-run after `docker compose down -v` (mounted via /docker-entrypoint-initdb.d/init.sql which only runs on FIRST container init for the postgres-data volume).

       CREATE SCHEMA IF NOT EXISTS destination;

       -- Users (literal hard-coded dev passwords — match defaults in .env.example; NOT real secrets)
       DO $$ BEGIN
         IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname='auth_svc') THEN
           CREATE ROLE auth_svc LOGIN PASSWORD 'auth_svc';
         END IF;
         IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname='trip_svc') THEN
           CREATE ROLE trip_svc LOGIN PASSWORD 'trip_svc';
         END IF;
         IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname='destination_svc') THEN
           CREATE ROLE destination_svc LOGIN PASSWORD 'destination_svc';
         END IF;
       END $$;

       -- Schema-scoped grants — per-service user has USAGE + CREATE on its own schema only (D-08 cross-schema denial)
       GRANT USAGE, CREATE ON SCHEMA auth TO auth_svc;
       GRANT USAGE, CREATE ON SCHEMA trip TO trip_svc;
       GRANT USAGE, CREATE ON SCHEMA destination TO destination_svc;

       -- search_path defaults — each user lands in its own schema by default
       ALTER ROLE auth_svc SET search_path = auth;
       ALTER ROLE trip_svc SET search_path = trip;
       ALTER ROLE destination_svc SET search_path = destination;
       ```
       **Note**: `IF NOT EXISTS` on `CREATE ROLE` is not standard PG syntax — hence the `DO $$ ... pg_roles WHERE rolname='X' ...` guard pattern. This makes re-execution after `docker compose down -v` (which wipes the volume and re-runs init.sql) safe and predictable.

    **Part A — Per-service multi-stage backend Dockerfiles (5 files).**

    Each backend service gets a multi-stage Dockerfile at `services/<svc>/Dockerfile`. The build context is the **repo root** (set in compose via `build.context: .` and `build.dockerfile: services/<svc>/Dockerfile`) — this is standard Spring Boot Gradle multi-module practice and is the only way to access the root `gradlew`, `settings.gradle.kts`, `build.gradle.kts`, `gradle/`, and shared `libs/` from within the build container.

    **CRITICAL**: Docker `COPY` does NOT support `../` parent-relative paths. The earlier draft of this plan included a parent-relative-COPY example — that approach is broken and is REMOVED here (WARNING 3 fix). The correct approach is `build.context: .` (repo root) so all paths in the Dockerfile are relative to the repo root.

    **Standard Dockerfile template** (each service substitutes its own `<svc>` and `<port>`):

    ```Dockerfile
    # Multi-stage backend Dockerfile — builds the bootJar INSIDE the container.
    # Build context: repo root (set in compose via build.context: . and build.dockerfile: services/<svc>/Dockerfile).
    # Satisfies ROADMAP SC#1 "no manual intervention" — no prior `./gradlew bootJar` required.

    # Stage 1: build with Gradle multi-module repo root as build context.
    FROM eclipse-temurin:21-jdk-alpine AS build
    WORKDIR /workspace
    # Install bash for the gradle wrapper (alpine ships with ash by default; gradlew works under ash but bash is safer).
    RUN apk add --no-cache bash
    # Copy build infrastructure first so dep-resolution layer is cached when only sources change.
    COPY gradle gradle
    COPY gradlew settings.gradle.kts build.gradle.kts gradle.properties ./
    # Make gradlew executable in case file mode was lost (e.g. via Windows checkout).
    RUN chmod +x gradlew
    # Copy shared libs (libs/observability, libs/error-handling, libs/api-contracts) — backend services depend on them.
    COPY libs libs
    # Copy ONLY this service's sources to keep cache-busting tight.
    COPY services/<svc> services/<svc>
    # Build only this service's bootJar; skip tests in the image build (tests run in CI).
    # The --no-daemon flag is recommended in containers (no daemon to reuse).
    # `--mount=type=cache` is a BuildKit feature; if BuildKit is unavailable the build still works just slower.
    RUN --mount=type=cache,target=/root/.gradle ./gradlew :services:<svc>:bootJar -x test --no-daemon

    # Stage 2: slim runtime.
    FROM eclipse-temurin:21-jre-alpine AS runtime
    WORKDIR /app
    # Install wget for healthcheck (alpine ships without it; busybox-wget is in alpine base).
    # eclipse-temurin:21-jre-alpine includes busybox wget — verify; if not, RUN apk add --no-cache wget.
    RUN apk add --no-cache wget
    COPY --from=build /workspace/services/<svc>/build/libs/*.jar app.jar
    EXPOSE <port>
    HEALTHCHECK --interval=10s --timeout=5s --retries=6 --start-period=20s \
      CMD wget -qO- http://localhost:<port>/actuator/health | grep -q '"status":"UP"' || exit 1
    ENTRYPOINT ["java","-jar","/app/app.jar"]
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host network ↔ compose-network services | D-22 dictates which ports are LAN-accessible vs loopback-only |
| compose internal network ↔ between containers | All inter-container traffic is on the compose bridge network; no auth in Phase 0 (Phase 1 adds JWT) |
| developer file system ↔ postgres data volume | `pgdata` named volume persists across restarts; `down -v` wipes |
| backend Dockerfile builder ↔ Maven Central / Gradle Plugin Portal | Multi-stage build pulls deps inside builder; supply-chain risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-00-31 | Information Disclosure | Postgres exposed to LAN with default `postgres/postgres` superuser credentials | HIGH | mitigate | D-22 binds postgres to `127.0.0.1:5432` only — not LAN-reachable. Per-service users exist with limited grants (D-08). Superuser is only used by init.sql and not by any service runtime. |
| T-00-32 | Privilege Escalation | Service connects with superuser credentials and modifies cross-schema objects | HIGH | mitigate | Per-service env vars (`AUTH_DB_USER` etc.) inject the per-service user; `application.yml` reads these env vars; init.sql grants `USAGE, CREATE` on each schema only to its owning user. Cross-schema write attempts fail with `permission denied`. |
| T-00-33 | DoS | Unhealthy service blocks all dependents indefinitely | MEDIUM | mitigate | All healthchecks have bounded `retries × interval` plus `start_period`. Worst case: service is marked unhealthy after ~50s; compose fails fast. `--wait-timeout 60` in smoke.sh enforces SC#1's bound. |
| T-00-34 | Tampering | Compose file edited to switch a downstream service to `0.0.0.0` binding, exposing it to LAN | MEDIUM | accept | Code review enforces D-22; no automated mitigation in Phase 0. Phase 10 may add a CI lint that fails the build if downstream services bind to `0.0.0.0`. |
| T-00-35 | Spoofing | Mailhog accepts SMTP from any container — but only on loopback (D-22) | LOW | accept | Mailhog SMTP port (1025) bound to 127.0.0.1; only the developer host can submit mail. Mailhog UI on 0.0.0.0:8025 is intentional for dev ergonomics. |
| T-00-36 | Tampering | postgres/init.sql tampered to grant `superuser` to a service user | HIGH | mitigate (review) | Code review; init.sql is in source control. The script as written grants only `USAGE, CREATE` on the service's own schema — no `superuser`, no `CREATEDB`, no `CREATEROLE`. |
| T-00-37 | Information Disclosure | Compose env injects raw DB passwords into container env (visible via `docker inspect`) | LOW | accept | Standard compose pattern; passwords are dev-only literal values matching `.env.example`. Production deployment (out of scope for v1) would use Docker Swarm secrets or Kubernetes secrets. |
| T-00-38 | Supply-Chain | Pulled images (`postgres:16`, `redis:7-alpine`, `eclipse-temurin:21-jdk-alpine`, `eclipse-temurin:21-jre-alpine`, etc.) tampered upstream | LOW | accept (Phase 10 owns) | Pull from official Docker Hub images; Phase 10 hardening can pin by digest. Multi-stage Dockerfiles use distinct builder + runtime base images, narrowing the attack surface in the runtime image. |
| T-00-52 | Supply-Chain | Multi-stage builder runs `./gradlew bootJar` which pulls Maven Central deps — tampered transitives could land in the runtime image | MEDIUM | mitigate | Gradle's dependency lockfile (Phase 10 hardening can enable `gradle/dependency-locks/`); the version catalog (Plan 01) pins every backend version literally. The runtime image only contains the bootJar (not the builder's `/root/.gradle` cache), so transitive metadata leaks are bounded. |
