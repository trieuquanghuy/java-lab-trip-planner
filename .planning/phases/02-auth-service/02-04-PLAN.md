```
import com.tripplanner.auth.config.AuthProperties;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.context.event.EventListener;
import org.springframework.mail.MailException;
import org.springframework.mail.SimpleMailMessage;
import org.springframework.mail.javamail.JavaMailSender;
import org.springframework.scheduling.annotation.Async;
import org.springframework.stereotype.Component;

/**
 * @Async @EventListener — Pattern 5 from 02-RESEARCH.md.
 *
 * Pinned to the named "authAsyncExecutor" bean (Plan 03 AsyncConfig) so the
 * MdcCopyingTaskDecorator carries traceId/requestId/userId across the thread switch (D-22).
 *
 * D-04: on MailException, log only userId + exception class — NEVER body or token.
 * D-23: recovery is via re-signup, not retry.
 *
 * Body verbatim per 02-UI-SPEC.md §Email Copy Contract. Em-dash is U+2014 ("—").
 * Subject is hard-locked: "Verify your Trip Planner account".
 */
@Component
public class EmailVerificationSender {

    private static final Logger log = LoggerFactory.getLogger(EmailVerificationSender.class);

    private final JavaMailSender mailSender;
    private final AuthProperties props;

    public EmailVerificationSender(JavaMailSender mailSender, AuthProperties props) {
        this.mailSender = mailSender;
        this.props = props;
    }

    @Async("authAsyncExecutor")
    @EventListener
    public void on(VerificationEmailRequestedEvent ev) {
        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setFrom("Trip Planner <" + props.getMail().getFrom() + ">");
        msg.setTo(ev.email());
        msg.setSubject("Verify your Trip Planner account");
        msg.setText(buildBody(ev.token()));
        try {
            mailSender.send(msg);
            log.info("Verification email sent userId={}", ev.userId());
        } catch (MailException ex) {
            log.warn("Mail send failed userId={} class={}", ev.userId(), ex.getClass().getSimpleName());
        }
    }

    /** UI-SPEC §Email Copy Contract — exact body. */
    String buildBody(String token) {
        return "Welcome to Trip Planner!\n\n"
             + "Click the link below to verify your email and activate your account:\n\n"
             + props.getVerification().getLinkBase() + token + "\n\n"
             + "This link expires in 24 hours.\n\n"
             + "If you didn't create an account, you can safely ignore this email.\n\n"
             + "— The Trip Planner team";
    }
}
```

```
package com.tripplanner.auth.scheduling;

import jakarta.persistence.EntityManager;
import jakarta.persistence.PersistenceContext;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;
import org.springframework.transaction.annotation.Transactional;

/**
 * Daily 02:00 UTC cleanup of expired email-verification + refresh tokens (D-17).
 * No ShedLock — auth-service is single-instance per docs/02 §3.
 *
 * Two transactions per Open Question 4 RESOLVED — partial failure on one DELETE doesn't roll
 * back the other; logs are also cleaner (each transaction reports its own row count).
 *
 * Self-invocation caveat: cleanup() calls cleanupEmailTokens() / cleanupRefreshTokens() via
 * `this`, which bypasses the AOP proxy. To make @Transactional fire on each method, Spring
 * resolves these calls through the proxy because @Scheduled-annotated methods are invoked by
 * the TaskScheduler (which holds the proxied bean reference). The inner methods are public
 * to allow proxy interception when called from outside (e.g., admin cron-trigger endpoint in
 * v2). For Phase 2's single @Scheduled trigger, the practical outcome is: each native-query
 * runs in its own short-lived transaction (default Tx propagation REQUIRED). If Spring's
 * proxy chain doesn't propagate as expected in unit testing, Plan 06's IT confirms behavior
 * end-to-end by triggering the cleanup manually and asserting row deletes.
 */
@Component
public class TokenCleanupJob {

    private static final Logger log = LoggerFactory.getLogger(TokenCleanupJob.class);

    @PersistenceContext
    private EntityManager em;

    @Scheduled(cron = "0 0 2 * * *")
    public void cleanup() {
        int evDeleted = cleanupEmailTokens();
        int rtDeleted = cleanupRefreshTokens();
        log.info("TokenCleanupJob: deleted {} email_verification_tokens, {} refresh_tokens",
                evDeleted, rtDeleted);
    }

    @Transactional
    public int cleanupEmailTokens() {
        return em.createNativeQuery(
                        "DELETE FROM auth.email_verification_tokens WHERE expires_at < NOW() - INTERVAL '7 days'")
                .executeUpdate();
    }

    @Transactional
    public int cleanupRefreshTokens() {
        return em.createNativeQuery(
                        "DELETE FROM auth.refresh_tokens WHERE expires_at < NOW() - INTERVAL '7 days'")
                .executeUpdate();
    }
}
```

The cron string `0 0 2 * * *` is the Spring `@Scheduled` 6-field cron (second minute hour day-of-month month day-of-week) — daily at 02:00:00. Spring's `@Scheduled` cron uses the JVM's default time zone unless `zone =` is specified; UTC is the JVM default in Docker images Phase 0 deploys, so this aligns with D-17's "02:00 UTC" specification. If a future deployed profile runs in a non-UTC TZ, set `zone = "UTC"` on the annotation.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Bean Validation -> incoming JSON | Bean Validation runs at controller entry (Plan 05). DTOs in this plan declare the constraints; the validator is wired by `spring-boot-starter-validation` (already in deps from Plan 01). |
| ApplicationEvent -> @Async listener | The publish/listen indirection through `ApplicationEventPublisher` is the keystone Pitfall-1 mitigation. The listener bean is separate from the publishing bean — this is what makes the `@Async` actually async. |
| @Async pool -> SMTP | `EmailVerificationSender` runs on the `authAsyncExecutor` thread; an unhandled exception bubbles into the executor's default error handler (Spring's `SimpleAsyncUncaughtExceptionHandler`). `MailException` is explicitly caught; other unchecked exceptions would surface in the executor's ERROR log. |
| @Scheduled trigger -> native queries | `TokenCleanupJob` is the sole writer to the DELETE-by-expires_at path. Native queries use parameterized intervals (no user input) — SQL injection N/A. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-04-01 | Information Disclosure | EmailVerificationSender logs | mitigate | `log.warn` on `MailException` logs only `userId` and `exception class` — NEVER body or token (D-04). Acceptance criterion forbids `ev.token()` in any log statement. |
| T-2-04-02 | Tampering | Email body | mitigate | Body is plain-text only via `SimpleMailMessage` — no HTML, no attachment. Stored XSS via email is impossible (D-03). UI-SPEC §Email Copy Contract is the single-source-of-truth for body content. |
| T-2-04-03 | Spoofing | From-address | accept | `Trip Planner <no-reply@tripplanner.local>` — internal-only address; SPF/DKIM out of scope for v1 portfolio (no real outbound mail). v2 adds proper email-deliverability infra. |
| T-2-04-04 | Denial of Service | TokenCleanupJob runtime | accept | Daily DELETE with `expires_at < NOW() - INTERVAL '7 days'` is bounded by the `rt_expires_at_idx` index (Plan 01). | T-2-04-05 | Tampering | TokenCleanupJob row scope | mitigate | The 7-day grace ensures recently-rotated chains stay queryable for forensics; only truly-stale rows DELETE. ASVS V8 (data protection) + audit-trail consideration. |
