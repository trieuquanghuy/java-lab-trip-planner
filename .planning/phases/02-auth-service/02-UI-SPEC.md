| Token placement | On its own line as a complete URL | D-03 (LOCKED) |

| Expiry note | "expires in 24 hours" (matches `email_verification_tokens.expires_at` 24h TTL per docs/05 §7) | D-03 (LOCKED) |

### Exact body (canonical — executor uses this verbatim)

```
Welcome to Trip Planner!

Click the link below to verify your email and activate your account:

http://localhost:8080/api/auth/verify?token={TOKEN}

This link expires in 24 hours.

If you didn't create an account, you can safely ignore this email.

— The Trip Planner team
```

**Implementation contract:**
- `{TOKEN}` is the 64-char hex token from `email_verification_tokens.token` (docs/05 §7).
- The gateway base URL prefix (`http://localhost:8080/api/auth/verify?token=`) is configurable via Spring property `app.verification.link-base` so docker / dev / future deployed profiles can override without code changes. Default value is the literal string above.
- Lines are separated by single `\n`. No trailing whitespace.
- The em-dash in the sign-off is `—` (U+2014). UTF-8 encoded.
- No marketing copy, no logo URLs, no unsubscribe link (all v2 / v9 polish).

**Non-functional notes:**
- The `traceId` from `@Async` MDC propagation (D-22) is NOT included in the email body — log-only.
- On `MailException`, the body is NEVER logged (D-04). Recovery is via re-signup (D-23).

## Server-Driven Copy Contract (Phase 2 ships this)

> The `detail` field of every RFC 7807 `ProblemDetail` response Phase 2 emits is user-facing copy. Phase 7's SPA will surface these strings (eventually localized in v2) in toasts and inline form errors. **The exact strings are locked here so the Phase 2 controller/service code uses them verbatim and Phase 7 can pre-write copy that matches.**

**Account-enumeration policy** (`docs/05 §9.1`): Every `detail` string must be generic enough that the response cannot be used to confirm account existence, password correctness, or token validity (beyond the public 24h expiry).

| Code | HTTP | `detail` string (canonical) | Notes |
|------|------|-----------------------------|-------|
| `auth.invalid_email` | 400 | `Invalid email format.` | Bean Validation `@Email` failure on signup or login (D-18). |
| `auth.weak_password` | 400 | `Password does not meet minimum requirements.` | `@Size(min=8)` on signup; never reveals min-length value to avoid policy fingerprinting. |
| `auth.email_already_registered` | 409 | **NOT EMITTED on `/api/auth/signup`.** Per D-23 (re-signup-of-unverified) and RESEARCH §Open Question 5 (recommendation Option A), signup of any existing email — verified or unverified — returns the same opaque `201 {userId}` envelope. Code stays in `docs/04 §6` catalog for future admin/internal endpoints but is not user-surfaced in v1 from auth-service. | Phase 2 plan should also raise this as the canonical resolution to RESEARCH §Open Question 5 before implementation. |
| `auth.invalid_credentials` | 400 | `Email or password is incorrect.` | Used for both unknown-email and wrong-password — generic per §9.1. |
| `auth.email_not_verified` | 403 | `Please verify your email before logging in.` | Login attempted before verify (security IT #7). |
| `auth.token_invalid` | 400 | `This verification link is invalid.` | Used for unknown / consumed / malformed verify tokens. (Per docs/05 §9.1, unknown and consumed share this code.) **Note:** the verify endpoint 302-redirects per D-02 — this `detail` string only surfaces in `application/problem+json` responses to non-redirect-capable callers (curl with `-i`, integration tests). Browser users see the FE `/verify?status=invalid` page. |
| `auth.token_expired` | 400 | `This verification link has expired.` | Verify token > 24h old. Same caveat as above (302 path is the user-facing surface). |
| `auth.refresh_invalid` | 401 | `Session expired. Please log in again.` | Bad / missing / revoked / rotated-and-replayed refresh token (security IT #5, #6). |
| `auth.rate_limited` | 429 | `Too many attempts. Please try again later.` | IP+email failed-login throttle (D-08, security IT #8). No `Retry-After` header in v1 (deferred per CONTEXT §Deferred). |
| `validation.failed` | 400 | `Request validation failed.` | Bean Validation generic fallback (D-18) when the failed field is neither `email` nor `password`. No field-name leak. |

**Implementation contract:**
- Every error response goes through `libs/error-handling/ProblemDetailFactory.of(status, code, detail)` (already shipped per Phase 0 plan 00-04).
- The strings above are the third argument (`detail`) verbatim. No string concatenation, no field-name interpolation, no traceId/userId/email substitution into the `detail` body.
- The `code` extension property (e.g. `auth.invalid_credentials`) is the stable machine-readable identifier the SPA keys off (per `docs/04 §2`). The SPA may use the `code` to render localized copy in v2 — the v1 `detail` strings above are the English source-of-truth.
- After BL-01 fix (D-14), `code` appears at JSON root (`$.code`), not `$.properties.code`. All Phase 2 ITs assert `$.code`.

**Title field** (RFC 7807 `title`): per `docs/04 §2` example, `title` is a short human-readable summary derived from the HTTP status (e.g. `"Bad Request"`, `"Unauthorized"`). Spring's auto-configured `ProblemDetail.forStatus()` already populates this — we do NOT override `title` per error code.

## Redirect Query-Param Contract (Phase 2 ships this)

> The verification endpoint is a `GET` that 302-redirects to the FE per D-02. The exact set of `?status=` values is locked here so the Phase 2 controller emits exactly these spellings and Phase 7 SPA can switch on them without surprises.

**Redirect target base:** `${app.frontend.base-url}/verify` (configurable via `FRONTEND_BASE_URL` env, default `http://localhost:5173/verify`).

| Query string | When emitted | Phase 7 SPA behavior (informational) |
|--------------|--------------|---------------------------------------|
| `?status=success` | Token was valid, unconsumed, unexpired. Backend has set `users.email_verified=true` and `email_verification_tokens.consumed_at=NOW()`. | Render success page with "Your account is verified" + "Log in" CTA. |
| `?status=invalid` | Token unknown, malformed, or already consumed. (Per docs/05 §9.1 — same code `auth.token_invalid` for unknown and consumed; the user does not learn which.) | Render generic "This link is invalid" page with "Sign up again" CTA. |
| `?status=expired` | Token was found in DB and unconsumed, but `expires_at < NOW()` (>24h since signup). | Render "This link has expired" page with "Request a new verification email by signing up again" CTA. |

**Spelling lock:** lowercase, single-word values. Exactly these three values — no `?status=ok` / `?status=bad` / `?status=timeout` variants. The auth-service controller emits them as `String` literals; Phase 7 SPA matches as `String` literals (no enum serialization in the wire contract).

**Other query params:** none. The token is consumed server-side (D-02) and never reaches the FE.

**Status code:** `HTTP 302 Found` with `Location: ${app.frontend.base-url}/verify?status=<value>`. (Spring `ResponseEntity.status(302).location(URI.create(...)).build()` is the idiomatic implementation per RESEARCH §Pattern 2.)

## Decision: Static Fallback Page

**SKIPPED.** No `verify-success.html` is shipped in auth-service.

**Rationale:**
- Per RESEARCH §Open Question 2 recommendation (and CONTEXT §Claude's Discretion bullet 3): the demo flow is `docker compose up` → FE is up at `:5173` → email link clicked → backend 302 → FE renders the canonical `/verify` page with shadcn / Tailwind / slate palette. If FE is down, the browser shows `ECONNREFUSED` — that's the demo's fault, not the auth-service's.
- A static fallback page in auth-service would have to render with inline CSS (no Tailwind / no shadcn at the Spring Boot static-resource layer), which would visually drift from the FE's design system on every Phase 7+ change. Maintenance cost for zero real-world recovery.
- Defense-in-depth recovery for users who hit the verify URL via copy-paste outside the demo: the 302 will still 302 (the redirect itself doesn't depend on the FE being up); only the final render fails. No data is lost.

**If reversed (user explicitly opts in):** the page would be a single static HTML file at `services/auth-service/src/main/resources/static/verify-success.html`, no inline images, no JS, three `<p>` tags with sans-serif body text and one `<a>` to `${app.frontend.base-url}` (env-substituted at build time via Spring resource filtering). This file is **not shipped** in v1.
